## 2. Scope Definition
*   **Problem Statement:** Users need to track reading progress (pending → reading → read) and record reviews only after finishing a book.
*   **Business Outcome:** Reading history visibility; personal ratings/opinions; foundation for future analytics/recommendations.
*   **In-Scope:** State transitions (pending/reading/read), rating (1-5), opinion (free text), timestamps (started/completed).
*   **Out-of-Scope:** Reading goals/challenges, social sharing, advanced review filtering.

## 3. Actors & Preconditions
*   **Primary Actors:** Authenticated user.
*   **Dependencies:** Personal Library (SPEC-0003), Identity & Access (SPEC-0001).
*   **Preconditions:** UserBook entry exists in user's library.

## 4. Functional Requirements
*   **REQ-001:** Update reading state on UserBook: PENDING → READING → READ (unidirectional).
*   **REQ-002:** Accept optional start_reading_date on state change to READING.
*   **REQ-003:** Accept optional completed_reading_date on state change to READ.
*   **REQ-004:** Create/update rating (1-5) and opinion (free text) only when state = READ.
*   **REQ-005:** Rating and opinion are nullable; opinion max 1000 characters.
*   **REQ-006:** Provide endpoints: PATCH /users/{id}/library/{book_id}/state, POST /users/{id}/library/{book_id}/review.
*   **REQ-007:** Retrieve review via GET /users/{id}/library/{book_id} (includes state, rating, opinion, timestamps).

## 5. Non-Functional Requirements
*   **NFR-001:** State update latency: <200ms.
*   **NFR-002:** Rating/opinion update atomic with respect to state (transactional).

## 6. Business Rules
*   **RULE-001:** Rating range enforced: 1-5 (enum or constraint); reject outside range.
*   **RULE-002:** Rating/opinion cannot be set if state != READ; validation on API gateway.
*   **RULE-003:** Opinion character count enforced at API + DB level; truncation not allowed (reject oversized).
*   **RULE-004:** State transitions immutable: if READ, user cannot revert to READING (history-only read).
*   **RULE-005:** Timestamps (start_reading_date, completed_reading_date) set by system (server time); client cannot override.

## 7. Acceptance Criteria
*   **AC-001:** Add book (state=PENDING) → change to READING with start_date → GET request returns state=READING, start_date set.
*   **AC-002:** State=READING → try to set rating=4 → receives 400 Bad Request (state is not READ).
*   **AC-003:** State=READ → set rating=4 and opinion="great book" → GET returns both fields; update succeeds.
*   **AC-004:** Opinion > 1000 chars → API rejects with 400 Bad Request.
*   **AC-005:** State transition PENDING → READING → READ all captured with timestamps; audit log reflects all changes.

## 9. Open Questions / Risks
*   **Q-001:** Should users be able to update completed_reading_date retroactively?
*   **Q-002:** Should rating/opinion be editable after initial submission?
*   **R-001:** State transitions not reversible; design UI to prevent accidental clicks.
