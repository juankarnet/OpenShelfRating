# Specification: SPEC-0008 - Remove Book from Personal Library

## 1. Metadata
*   **Spec ID:** SPEC-0008
*   **Version:** 1.0.0
*   **Status:** Implemented
*   **Owner:** BA Agent
*   **Source Milestone:** Personal Library Operations
*   **Last Updated:** 2026-07-08
*   **Implementation Status:** Backend Implemented; Web Frontend Implemented; Tests pending

## 2. Scope Definition
*   **Problem Statement:** A user who added a book by mistake or who no longer wants to track it in their library has no documented, formally specified path to remove it. The feature exists in code but lacks a governing specification and acceptance-tested behaviour contract.
*   **Business Outcome:** Users can remove any active book from their personal library via a confirmed destructive action; the book record is preserved in the system (soft-delete) and can be re-added later.
*   **In-Scope:** Single-book removal from personal library, soft-delete semantics, confirmation UX, error feedback, cache invalidation, re-add behaviour contract.
*   **Out-of-Scope:** Bulk removal, permanent hard-delete, admin-initiated removal of another user's library entry (deferred), undo/restore within time window (deferred).

## 3. Actors & Preconditions
*   **Primary Actors:** Authenticated user (owner), system administrator.
*   **Dependencies:** Personal Library Management (SPEC-0003), Identity & Access (SPEC-0001).
*   **Preconditions:** User is authenticated; the book is present in the user's active library (`deleted_at IS NULL`).

## 4. Functional Requirements

### 4.1 Backend
*   **REQ-001:** Provide endpoint `DELETE /users/{userId}/library/{bookId}` that performs a soft-delete on the matching active `UserBook` record.
*   **REQ-002:** Only the library owner (`pathUserId == principalUserId`) or an admin may invoke the endpoint; any other principal receives 403 FORBIDDEN.
*   **REQ-003:** If no active `UserBook` exists for the given `(userId, bookId)` pair, respond 404 NOT FOUND.
*   **REQ-004:** On success respond 204 No Content (empty body).
*   **REQ-005:** Soft-delete sets `deleted_at = now()`; no row is physically removed; the record is reachable with `includeDeleted=true` on the list endpoint (SPEC-0003 REQ-006).
*   **REQ-012:** Provide endpoint `GET /books/{bookId}/deletion-eligibility?actorUserId=...` returning whether the actor can delete the book from system catalog after unlinking from personal library.
*   **REQ-013:** Eligibility must be `true` only when the actor is the creator of the book and there are zero active `user_books` links for that `bookId`.
*   **REQ-014:** Provide endpoint `DELETE /books/{bookId}?actorUserId=...` to delete the book from catalog only when REQ-013 is satisfied; otherwise return 403/409 as appropriate.

### 4.2 Frontend
*   **REQ-006:** Each book card in both Dashboard and Library list views exposes a contextual menu (⋮) containing a "Remove" action styled as a danger item.
*   **REQ-007:** Clicking "Remove" opens a `ConfirmActionModal` with title "Remove Book" and message containing the book title, before any API call is made.
*   **REQ-008:** On user confirmation, call the remove mutation; disable the confirm button while the request is in-flight.
*   **REQ-009:** On success, display a visible notification (toast or inline message) confirming the removal; library list and stats must refresh automatically.
*   **REQ-010:** On failure, display a descriptive error message (not just "Failed to remove book") so the user understands what went wrong.
*   **REQ-011:** Also expose a visible remove action from BookDetailModal when opened from either Dashboard or Library views.
*   **REQ-015:** After successful unlink (`REQ-001`), call deletion eligibility endpoint; if eligible, show a second confirmation modal asking whether to delete the book from system catalog.
*   **REQ-016:** The second modal must only appear for creator-owned books with zero active links and must trigger `DELETE /books/{bookId}` only on explicit user confirmation.

## 5. Non-Functional Requirements
*   **NFR-001:** Soft-delete operation must complete in <200 ms (single row update + index).
*   **NFR-002:** Frontend confirmation modal must not block the main UI thread.
*   **NFR-003:** After removal, `GET /users/{id}/library` must exclude the deleted record by default within the same request cycle.

## 6. Business Rules
*   **RULE-001:** Soft-delete preserves full audit trail (`deleted_at` timestamp, all reading history fields). Hard-delete is out of scope for MVP.
*   **RULE-002:** Re-adding a soft-deleted book reactivates the original `UserBook` record (resets state to PENDING and clears reading dates, rating, opinion). Users are not warned about data loss in the current implementation; documenting this as a known gap.
*   **RULE-003:** The confirmation modal is mandatory; removal cannot be triggered in a single click/tap.
*   **RULE-006:** Catalog deletion is a separate, explicit second confirmation and never implicit from library unlink.
*   **RULE-004:** Only one active entry per `(userId, bookId)` is permitted (enforced by partial unique index from SPEC-0003 RULE-001); this constraint remains valid post-delete since `deleted_at IS NOT NULL` rows are outside the unique constraint.
*   **RULE-005:** Removal is non-reversible from the user-facing UI in MVP (no restore/undo). Re-adding is the recovery path.

## 7. Acceptance Criteria
*   **AC-001:** Authenticated owner calls `DELETE /users/{id}/library/{bookId}` for an active book → response 204, book excluded from subsequent `GET /users/{id}/library` (default).
*   **AC-002:** Authenticated owner calls `DELETE` with `includeDeleted=true` on list endpoint after removal → deleted book still appears with non-null `deletedAt`.
*   **AC-003:** Non-owner, non-admin calls `DELETE` → response 403 FORBIDDEN.
*   **AC-004:** `DELETE` called for bookId not in user's active library → response 404 NOT FOUND.
*   **AC-005:** After removal, `GET /users/{id}/library/stats` reflects decremented book counts.
*   **AC-006:** UI: Remove menu item visible on each BookCard in Dashboard and Library pages; clicking it opens confirmation modal before any request.
*   **AC-007:** UI: Confirm button disabled during in-flight request.
*   **AC-008:** UI: Success notification visible after confirmed removal; library list refreshes without full page reload.
*   **AC-009:** UI: Network error during removal → descriptive error displayed without dismissing the modal silently.
*   **AC-010:** Owner removes book, then re-adds the same book → original `userBookId` is reused, no duplicate, reading state reset to PENDING.
*   **AC-012:** If user is creator and no active links remain after unlink, UI shows a second confirmation to delete book from system catalog.
*   **AC-013:** If second confirmation is accepted, `DELETE /books/{bookId}` returns 204 and book no longer appears in catalog search.
*   **AC-014:** If user is not creator or active links remain for other users, second confirmation must not be shown.
*   **AC-011:** UI: BookDetailModal displays a visible remove button in Dashboard and Library contexts; action triggers the same confirmation flow.

## 9. Open Questions / Risks / Decisions
*   **Q-001:** Should the UI warn users that re-adding the book later will reset all reading history and ratings?
    - **Tentative answer:** Yes — add warning text inside the confirmation modal body (deferred to UX polish sprint).
*   **Q-002:** Should admins be able to remove books from other users' libraries from an admin panel?
    - **Answer:** Out of scope for MVP. Authorization model already supports admin principal; endpoint is accessible to admins.
*   **R-001:** Data loss on re-add — users not warned that reactivation resets state and rating. Medium risk; add warning text to confirmation modal.
*   **R-002:** Frontend error messages are currently generic ("Failed to remove book"). Should propagate server error message. Captured in REQ-010.

## 10. Known Gaps (Implementation Delta)
*   **GAP-001 (Low):** End-to-end automated coverage for remove visibility/confirmation in Dashboard and Library flows is still pending.
