# Specification: SPEC-0005 - Media Management (Avatars & Book Covers)

## 1. Metadata
*   **Spec ID:** SPEC-0005
*   **Version:** 1.2.0
*   **Status:** In Progress (backend permissions aligned; frontend editable modal flow pending)
*   **Owner:** BA Agent, Architect Agent
*   **Source Milestone:** Functional Alignment
*   **Last Updated:** 2026-07-08

## 2. Scope Definition
*   **Problem Statement:** Users need to upload and manage profile avatars; books need cover images for visual catalog browsing.
*   **Business Outcome:** Personalized profiles; visual book identification; improved UX.
*   **In-Scope:** Upload avatar (user), upload cover (book), retrieve via URL, replace/delete images.
*   **Out-of-Scope:** Image cropping/resizing in backend, CDN optimization, image analytics.

## 3. Actors & Preconditions
*   **Primary Actors:** Authenticated user (avatar), end user or admin (book cover).
*   **Dependencies:** S3-compatible storage (MinIO/AWS S3), Identity & Access (SPEC-0001), Book Catalog (SPEC-0002).
*   **Preconditions:** S3 bucket initialized, user/book record exists.

## 4. Functional Requirements
*   **REQ-001:** Accept image upload (JPG, PNG, WebP) with max size 5MB for avatars, 10MB for covers.
*   **REQ-002:** Validate MIME type server-side; reject invalid formats.
*   **REQ-003:** Store image in S3 with path: `/avatars/{user_id}/{timestamp}.{ext}` or `/covers/{book_id}/{timestamp}.{ext}`.
*   **REQ-004:** Return presigned GET URL (24h expiry) for retrieval from dedicated media GET endpoints; store stable object path reference in user/book record.
*   **REQ-005:** Provide endpoints: POST /users/{id}/avatar, GET /users/{id}/avatar, DELETE /users/{id}/avatar, POST /books/{id}/cover, GET /books/{id}/cover, DELETE /books/{id}/cover.
*   **REQ-006:** Allow creator or admin to replace/delete a book cover; new upload creates a new immutable S3 path, old object retained and soft-deleted in audit table.
*   **REQ-007:** Soft-delete: mark image deleted_at; retrieval returns placeholder until new upload.
*   **REQ-008:** The web frontend must expose a book detail view as an in-page modal (popup) accessible from the library book card; the modal displays the full book metadata (title, author, ISBN, publisher, language, pages, publication date, cover image).
*   **REQ-009:** When the authenticated user is the original creator of the book (`book.createdBy == currentUser.userId`) or has admin role, the book detail modal must expose cover management actions (upload/replace/delete according to current cover state) and refresh the displayed cover without closing the modal.

## 5. Non-Functional Requirements
*   **NFR-001:** Upload latency: <2s; image size validation at streaming boundary.
*   **NFR-002:** Presigned URL generation latency: <100ms.
*   **NFR-003:** Image retrieval caching: Set Cache-Control: public, max-age=86400 (24h).

## 6. Business Rules
*   **RULE-001:** Allowed MIME types: image/jpeg, image/png, image/webp; reject all others.
*   **RULE-002:** File size limits: avatar ≤5MB, cover ≤10MB; reject oversized uploads.
*   **RULE-003:** Avatar upload/replace/delete allowed to owner or admin. Book cover upload/replace/delete allowed to the book creator or an admin.
*   **RULE-006:** The book detail modal renders cover management actions when the current user is the book creator or an admin; non-authorized users see the cover in read-only mode.
*   **RULE-004:** Image deletion: soft-delete only; original S3 object retained for audit.
*   **RULE-005:** S3 paths are immutable; new uploads generate new paths with timestamp.

## 7. Acceptance Criteria
*   **AC-001:** Upload avatar (JPG, 2MB) → returned presigned URL valid for 24h; GET request returns image.
*   **AC-002:** Upload cover (PNG, 8MB) → stored and linked to book record; GET /books/{id} returns cover_url.
*   **AC-003:** Upload avatar (BMP) → API returns 400 Bad Request (unsupported format).
*   **AC-004:** Upload avatar (15MB) → API returns 413 Payload Too Large.
*   **AC-005:** Replace avatar → new file stored, old URL removed from user record, GET returns new image.
*   **AC-006:** Non-owner tries to upload avatar for another user → 403 Forbidden.
*   **AC-007:** User opens a book card in library → detail modal appears with full book metadata and cover image (or placeholder).
*   **AC-008:** Creator of a book without cover opens detail modal → cover upload control is visible; after uploading, cover appears in the modal and in the library card without page reload.
*   **AC-009:** Creator of a book with an existing cover opens detail modal → replace/delete controls are visible and both operations succeed.
*   **AC-010:** Non-creator/non-admin opens detail modal of a book already with cover → no management controls are shown (cover is display-only).

## 9. Open Questions / Risks
*   **Decision Log (2026-06-26):**
	- Retrieval contract uses JSON payload with `presignedUrl` + `expiresAt` (no redirect).
	- API authorization uses authenticated principal as source of truth; development profile keeps controlled fallback behavior when principal is unavailable.
	- Book cover permissions split by lifecycle: create (any authenticated user if no cover), replace/delete (admin only).
	- Avatar and cover DELETE endpoints are included in scope.
*   **R-001:** S3 availability issues block uploads; implement retry + local fallback temp storage.
*   **R-002:** Presigned URL leakage if shared; consider short expiry or one-time tokens in Phase 2.
